# HJ 金物流與材積判斷補充

> 日期：2026-05-01
> 依據：綠界 ECPay Developers / 綠界物流服務手冊官方文件
> 目的：釐清「商品材積大小能否運送」是由金物流自動判斷，還是馬亞需要先寫規則判斷。

---

## 1. 結論

若 HJ 所說的「金物流」是綠界 ECPay 金物流，判斷結論是：

**材積 / 重量是否可使用超商或宅配，網站端應該先自行依規則判斷。不能假設金物流 API 會替網站完整判斷並回傳可用配送方式。**

綠界物流 API 的主要責任是：

- 建立物流訂單。
- 建立暫存物流訂單 / 正式物流訂單。
- 選擇門市。
- 列印託運單。
- 查詢物流訂單。
- 接收物流狀態通知。
- 建立逆物流訂單。

材積與重量限制則是各物流商規範。綠界文件要求賣家寄件前自行確認包裹尺寸與重量符合通路規範；超重或超尺寸包裹無法受理寄件。

---

## 2. 為什麼不能交給金物流自動判斷

### 2.1 超商 API 建單參數沒有完整長寬高

綠界舊版超商物流訂單建立 API 主要傳：

- 物流類型 `LogisticsType`
- 物流子類型 `LogisticsSubType`
- 商品金額 `GoodsAmount`
- 代收金額 `CollectionAmount`
- 商品名稱 `GoodsName`
- 寄件人 / 收件人資料
- 收件門市
- 回傳 URL

文件中未看到讓網站傳入每個包裹長、寬、高、總重量，並由綠界回傳「這張訂單可不可超商」的完整判斷欄位。

### 2.2 宅配 API 有重量 / 規格欄位，但仍不是自動包裝計算

宅配 API 有：

- `GoodsWeight`：商品重量，郵局必填，上限 20kg，小數 3 位，單位公斤。
- `Specification`：規格，黑貓可填 60cm / 90cm / 120cm / 150cm。

這代表宅配建單可以傳重量與規格級距，但「多商品加總後應該選哪個規格」、「箱購 + 條購如何合併包裝」、「是否能超商」仍需要網站先算好。

### 2.3 綠界官方文件把尺寸 / 重量列為寄件前檢查項

綠界物流服務手冊把「包裹尺寸與重量符合通路規範」列為寄件前須檢查事項，並說不同物流通路商有不同尺寸與重量上限，超重或超尺寸包裹無法受理寄件。

因此比較安全的設計不是「先讓客戶選超商，送 API 失敗再改宅配」，而是「網站在結帳前就判斷可用配送方式」。

## 3. HJ 應採用的系統責任分工

### 3.1 網站端 / 馬亞需要做

1. 商品資料需有材積 / 重量欄位。
2. 商品需能判斷箱購、條購、箱購 + 條購。
3. 購物車需加總：
   - 總重量。
   - 包裹尺寸或材積。
   - 是否含不可超商商品。
   - 是否需要拆包。
4. 結帳頁需依規則顯示可用配送方式：
   - 可超商：顯示超商選項。
   - 不可超商：隱藏超商，只顯示宅配或自取。
   - 離島：提示聯絡客服或依 HJ 規則處理。
5. 宅配運費需依 HJ 的「才數 / 材積級距 / 免運門檻」計算。
6. 超商運費需依 HJ 的超商門檻與限制計算。
7. 建立金物流訂單前，系統需已選定正確物流方式。

### 3.2 金物流負責

1. 建立物流訂單。
2. 取得超商門市或開啟物流選擇頁。
3. 列印託運單。
4. 回傳物流單號 / 寄貨編號。
5. 通知物流狀態。
6. 處理逆物流（若 HJ 申請的物流型態支援）。

### 3.3 凌越 ERP 負責

1. 提供商品重量 / 材積來源欄位。
2. 記錄訂單、銷貨、出庫資料。
3. 可視需求記錄物流公司、物流單號或出庫狀態。

凌越 ERP 不應被視為超商材積判斷引擎；它可提供商品資料，真正的配送判斷仍在網站。後續問凌越時只確認資料欄位與回寫欄位，不把「能不能配送」這個規則決策推給凌越。

## 4. 需要再跟 HJ 確認的材積規則

請 HJ 補明確規則，馬亞才寫得出正確判斷：

1. 每個 SKU 是否都會填：
   - 單條長寬高。
   - 單條重量。
   - 整箱長寬高。
   - 整箱重量。
   - 才數。
2. 「才」如何換算？是 HJ 自行填入結果，還是系統依長寬高計算？
3. 多品項混合下單時：
   - 材積直接加總？
   - 依包裹最大長寬高重新估算？
   - 需要人工判斷是否拆包？
4. 箱購 + 條購同時下單時，是否完全依 HJ 提供的運費圖？
5. 若一張訂單超過超商限制，是否只顯示宅配？
6. 若一張訂單宅配也超過單件限制，是否：
   - 拆成多件。
   - 提示客服聯繫。
   - 禁止結帳。
7. 四大超商限制是否全部用 HJ 回覆的 45 x 30 x 30 cm / 5kg，還是要依綠界各通路最新規範分別處理？

---

## 5. 需要馬亞技術確認的點

1. 結帳前是否要做「配送方式判斷服務」。
2. 材積規則是否做成後台可維護，不寫死在程式。
3. 商品是否需維護「不可超商」旗標。
4. 若材積資料缺漏，前台應：
   - 禁止上架。
   - 禁止超商。
   - 仍允許但提示管理員。
5. 是否需要支援拆包；若要拆包，運費與物流單號會變複雜。
6. 金物流建單失敗時，訂單狀態如何處理。
7. 若 HJ 使用綠界新版全方位物流服務，是否採用暫存物流訂單流程，出貨時才建立正式物流訂單。

---

## 6. 初步建議

第一階段建議採保守做法：

1. 網站先完成材積 / 重量 / 箱條規則判斷。
2. 結帳頁只顯示已通過判斷的配送方式。
3. 超商若超過限制，直接不顯示超商選項。
4. 宅配依 HJ 運費表計算。
5. 金物流只負責建單、列印託運單、物流狀態，不負責決定可不可配送。
6. 暫不做自動拆包，除非 HJ 明確要求並提供拆包規則。
