## 1. 同意 / 不同意 Codex 主要結論

3. **價格規則是核心未知**：HJ 「北部直客／中南部直客／北部盤商／中南部盤商」+ 「每商品設定不同等級價」可能超過 5 組價，必須請凌越確認是否有客戶別商品價格表 / 客戶貨品項號對照 API。我索引 §3.1 也標了 SK_PRICE1~5 + SK_FPRICE1~5 = 5 + 5 = 10 組，但這是「同商品 5 個價格層級 + 5 個輔助單位的對應價」，不是「10 種會員等級」。Codex 的擔憂正確。

5. **私版詢價 → 客戶報價單可對應，但時機點未知**：對。`0000A5` 結構可承接，但「網站建立詢價當下進 ERP 還是 LINE 確認後才進」是 HJ 業務決定。
6. **超商寄件單不在凌越 API**：對，需另接綠界物流 / 物流商。
7. **金流退款與發票責任三方分工（綠界/網站/凌越）**：分工合理。
8. **ECPay 物流補充文件的結論「材積判斷由網站做，不能交給金物流」**：完全同意，且綠界官方文件確實把尺寸/重量列為「寄件前自行檢查」，這條結論很關鍵，避免後續整合時責任歸屬混亂。
9. **凌越邊界（不問 SEO、文案、上下架、前台流程）**：邊界劃得對。

### 1.2 ⚠ 部分同意 / 需補充

1. **「未付款訂單是否先進凌越」這題的優先級**：Codex 標為高優先「不確認會卡架構」是對的，但建議再補一個前置題給 HJ：**新會員第一次下單時，可能還沒有 ERP 客戶編號（CT_NO）**，但 `OR_CTNO` 是必填。這個情境必須先決定：
   - A. 網站訂單同步 ERP 前，先呼叫 ERP 建客戶 → 拿到 CT_NO → 再建訂單。
   - B. 用一個「待綁定」佔位 CT_NO，事後人工配對。
   - C. 網站訂單先 pending，等管理員審核並綁 CT_NO 後才同步。
   這影響「下單失敗率」與「客服工作量」，Codex 目前的 hj-maya-to-hj 清單沒明確問到。建議補。

2. **`hj-maya-to-lingyue-api-confirmation-items.md` §11.6 銷退觸發發票作廢**：Codex 有問「`0000A2` 是否可觸發發票作廢」，但我從索引 §3.6 看到主檔記載 0000A2 的 imode 第 12 位有「轉入退貨單時，如果有對應銷貨單其發票號碼自動處理作廢」（內定 0=否）。這代表凌越其實已支援這個旗標，所以問法可以更精準：**「imode 第 12 位是否就是 HJ 退刷時要打開的旗標？打開後雲端電子發票會同步作廢嗎？還是只作廢 ERP 內部記錄？」**。

3. **訂單同步建議的 `imode` 第 1 碼**：Codex 在 §7.3 問「imode 第一碼建議用哪種處理」是對的，但建議加入：**HJ 訂單編號規則建議由網站主導（用 `imode=0` 跳過存在的 + 失敗時前綴流水重試）**，而不是凌越續編。理由：網站訂單編號要在客戶訂單成立頁立刻顯示，不能等待 ERP 回傳。

4. **`hj-maya-to-hj-confirmation-items.md` §6 庫存問題**：Codex 沒問到「HJ 預留給連鎖店的數量是用『客戶別保留』還是『獨立倉庫』實作」。從凌越 API 看，倉庫 `000004` + 出庫 `0000B7` 都支援多倉，**最乾淨的做法是給連鎖店一個獨立倉庫，網站只查一般倉庫的可用量**。建議加一題給 HJ：「預留庫存是否可實作為獨立『連鎖店倉』，網站只查一般倉？」

### 1.3 ❌ 不同意 / 反對

無重大反對。Codex 的判斷整體保守、可控，沒有過度承諾。

## 5. 需要馬亞技術判斷的事項（補充）

對應 `hj-followup-questions-customer-maya-lingyue.md` §2，建議補：

### 5.1 整體架構（補 #5）

> **建議馬亞採用「整合服務層 / BFF」模式，把凌越 SOAP 包成內部 REST / GraphQL：**
> - 整合層管金鑰快取（30 秒短 TTL）+ 自動續金鑰
> - 重試策略（金鑰失效自動重試一次、其他錯誤依錯誤碼分流）
> - 錯誤訊息正規化（凌越 -1 ~ -5 → 業務語意錯誤）
> - 觀測：每次呼叫紀錄 idakd / imode / 回傳碼，方便除錯
> 這層做好之後，前台 / 後台都不直接碰 SOAP，未來若凌越改 API 也只改一處。

### 5.2 訂單同步（補 #6）

> **建議用 Outbox Pattern 而不是直接同步呼叫：**
> - 網站訂單成立 → 寫入訂單表 + 寫入 outbox 表（pending）
> - 背景 worker 拉 outbox → 呼叫凌越 → 更新狀態（synced / failed）
> - 失敗自動重試（指數退避，3 / 30 / 300 秒）+ 超過上限通知管理員
> - 客戶結帳體驗不被 ERP 影響（凌越掛了仍可下單）
> 這個 pattern 解 §1.4 「訂單同步失敗時客戶端如何處理」的最佳答案。

### 5.3 庫存策略（補 #1 → 具體化）

> **建議三段式策略：**
> - 商品列表頁：讀網站庫存快取（TTL 5-10 分鐘）
> - 商品詳情頁：讀網站庫存快取 + 觸發背景刷新
> - 加入購物車 / 結帳前：lock 查凌越 `000009`，可用量 `WD_AMTOK` < 訂購量則擋下
> - 訂單建立成功後：本地扣減快取，等下次同步以凌越為準
> 這個策略解「即時性」與「ERP 負載」的平衡。

### 5.4 LINE 帶詢價編號（補 §G）

> **若採 LINE Messaging API + 已綁定**：可直接 push 詢價摘要到客戶 LINE 一對一。
> **若採加好友 + 帶編號（保底方案 A）**：用 LINE URL Scheme `https://line.me/R/oaMessage/{LINE_ID}/?text={URL_ENCODED_TEXT}` 預填訊息，客戶點按即送出。馬亞需驗證最新 LINE 規範是否仍支援此 schema（過去幾年 LINE 改過幾次）。
