## 2. 回覆 Claude Q1-Q6

### Q1. 問題分層怎麼分

#### B6 物流 / 超商寄件單 / 材積

我建議「超商寄件單是否要批次列印」列初階，這點你已標對，因為它會直接影響後台範圍與是否另接金物流。

「箱購 + 條購材積加總公式」不必在提案階段問到公式細節，但應該在初階問一個粗粒度問題：

> HJ 是否已經有一套判斷超商 / 宅配可配送的材積與重量規則？若沒有，是否接受由網站採保守規則：超過限制就只顯示宅配或聯絡客服？

也就是說，**公式細節留 kickoff，規則是否存在與是否接受保守判斷要初階問**。

#### B11 訂單狀態 source of truth

這題不建議用「source of truth」丟給 HJ。客戶不需要決定資料庫架構。

提案階段應改問業務語言：

> 訂單狀態是由 HJ 人員在網站後台更新，還是希望凌越狀態異動後自動反映到網站？

技術上我仍建議網站維護前台狀態，凌越作 ERP 單據來源與可同步來源。這個方向可以由馬亞內部判斷，不要讓 HJ 在「網站 / ERP 誰是主資料」這種技術詞上做決策。

#### B3 會員價格類別

這題是混合題，但提案階段先給 HJ 問業務版：

> 會員價格類別總共有幾種？是否會有單一客戶專屬價？

如果 HJ 回答「只有固定幾種」，再請凌越確認 5 組價是否足夠。如果 HJ 回答「有客戶專屬價」，就必須問凌越是否有客戶別商品價格 API。

分層上：

- HJ 初階：價格類別與是否有客戶專屬價。
- 凌越提案階段：是否有客戶別商品價格表 API。
- kickoff：欄位、XML、同步方向。

### Q2. 給凌越的 3 題是否太精簡

我覺得 3 題方向對，但偏少。建議提案階段問 4 題：

1. HJ 帳號目前授權哪些 API 資料類別？是否包含商品、客戶、訂單、庫存、收款、發票、報價。
2. 是否有客戶別商品價格表 / 客戶貨品項號對照 API？若有，是否可提供欄位或簡例。
3. 簽約後多久可提供測試環境、endpoint / WSDL、帳密、公司代號與核心流程 XML 範例。
4. 凌越是否有固定技術窗口或既有第三方串接流程？HJ 是否曾有第三方系統對接經驗？

我不建議提案階段直接問「SLA」。這個詞太正式，容易讓凌越覺得已進入合約維運談判。改成「技術窗口 / 對接流程 / 既有串接經驗」比較自然。

Sandbox 可併入第 3 題，不必獨立成一題。

---

### Q3. BFF / Outbox / 三段式庫存要不要對外提

不要對外講技術名詞，但可以在提案文件露一句業務語言，顯示有備課。

建議對外寫：

> ERP 串接會採背景同步、失敗重送與人工補送機制，避免 ERP 暫時無法連線時影響客戶完成下單。

不要寫：

- BFF
- Outbox Pattern
- REST / GraphQL
- 指數退避
- 金鑰快取

這些留在內部架構備忘。對 HJ 來說，他們只需要知道「不會因 ERP 一時連不上就讓客人卡在結帳」。

### Q4. 新會員首單 CT_NO 策略要不要入提案題

同意列入初階，但不要用 `CT_NO` 這個欄位名問 HJ。

建議併入「訂單同步」主題，問法如下：

> 新會員第一次下單時，若凌越還沒有這位客戶資料，HJ 希望網站先自動建立凌越客戶後再送訂單，還是先由網站成立訂單，待管理員審核 / 綁定客戶編號後再同步凌越？

我不建議預設由我們內部直接採 A。因為自動建立 ERP 客戶會影響 HJ 客戶主檔品質，也可能讓 ERP 出現很多未審核客戶。這是業務與資料治理問題，應讓 HJ 知道並拍板。

所以我的答案是：**同意初階，但包進訂單同步題，不另外增加成第 11 題。**

---

### Q5. 庫存延遲 5-10 分鐘這個數字 OK 嗎

可以保留，但要用「例子」而不是「建議承諾」。

建議寫法：

> HJ 對網站庫存更新的期待是秒級、幾分鐘內，例如 5-10 分鐘，還是只要結帳前再次確認準確即可？

這樣有三個好處：

1. 給 HJ 一個想像錨點，不會只回答「要即時」。
2. 不把 5-10 分鐘包裝成我們已承諾的 SLA。
3. 可以自然導向成本差異：秒級通常較貴、幾分鐘內較穩、結帳前準確成本最低。

如果正式提案文字要更穩，可以寫：

> 初步建議以「商品頁快取、結帳前再次查詢 ERP 可用量」為第一階段方案；若 HJ 要求秒級同步，需請凌越確認是否支援推播或加開機制。

---

### Q6. 退貨連動發票作廢要不要在提案問

不建議把 `imode 第 12 位` 或「自動作廢旗標」丟給 HJ。這是 kickoff 後問凌越 / 會計的技術細節。

但提案階段可以保留一個業務版問題，放在金流 / 發票主題底下：

> 退貨退款時，發票作廢目前由會計人工處理，還是希望系統與退貨流程連動？

這樣不會太技術，但能抓範圍。如果 HJ 說要連動，就知道要把發票作廢、綠界退刷、凌越銷退單一起納入後續規格；如果 HJ 說會計人工，第一階段可以先簡化。

所以我的答案是：**提案問業務版，技術版留 kickoff。**

## 5. 內部架構備忘清單

以下不要在提案階段丟給 HJ 或凌越，但馬亞與我們內部要先有共識：

1. SOAP 外包一層整合服務，避免前後台直接碰凌越 SOAP。
2. 凌越 pass key 30 秒失效，整合層要處理短期快取與自動重取。
3. 訂單同步採背景推送 + 失敗重送 + 人工補送，不讓 ERP 暫時失聯阻塞客戶結帳。
4. 庫存採三段式：列表頁快取、詳情頁快取加背景刷新、結帳前查凌越可用量。
5. 網站維護前台訂單狀態，凌越狀態只作同步來源或管理依據。
6. 網站訂單編號建議由網站主導，凌越單號 / 回傳結果另記錄。
7. 金物流只建單 / 標籤 / 貨態；材積與可配送判斷在網站做。
8. 退貨退款與發票作廢要拆成業務流程、金流退刷、凌越銷退、發票作廢四段。
