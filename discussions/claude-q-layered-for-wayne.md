## A. 直接回答 Wayne 的三個原始問題

### A1. 這份凌越 API 文件夠讓馬亞直接做嗎？

**❌ 不夠。可作為開發起點，但無法直接開工。**

| 夠的部分 | 差的部分（必須由凌越補件） |
|---|---|
| 知道凌越是 SOAP Web Service | 實際 endpoint / WSDL |
| 5 個 API 函式（取金鑰、轉入、轉出、刪除、分頁） | 測試環境 + 帳密 + 公司代號 |
| ~50 個資料類別、欄位名稱、XML 結構 | HJ 帳號實際授權哪些資料類別（每家公司不同） |
| imode 30 BYTES 參數規則 | HJ 真實欄位對照（凌越通用文件 vs HJ 客製） |
| 大致的錯誤碼分類 | 各流程的可執行 XML 範例 |
| | 必填欄位、預設值、單號規則 |
| | 錯誤碼細節與重送條件 |
| | 是否有「客戶別商品價格表」API（會員價超過 5 組時關鍵） |

**提案說法建議**：「凌越 API 文件方向可行，HJ 端有完整資料結構支援電商整合。簽約後 kickoff 第 N 個工作天內，凌越需補測試環境 + XML 範例 + 欄位對照表，再進入實作。」

### A2. 合作後要跟凌越補什麼？

四大類，依優先序：

1. **環境與權限**：endpoint、WSDL、測試帳密、公司代號、HJ 授權資料類別清單
2. **欄位實際樣貌**：HJ 商品 / 客戶 / 訂單欄位的真實 XML 範例
3. **流程細節**：訂單編號規則、未付款訂單處理、錯誤碼、重送策略
4. **特殊需求**：客戶別商品價格 API、超商寄件單支援度、庫存推播能力

完整清單在 `hj-maya-to-lingyue-api-confirmation-items.md`（80+ 題），**提案階段不該全送**（見 §C2）。

### A3. 客戶要的「ERP 即時更新」可行嗎？

**部分可行，但「即時」要重新定義**。

| HJ 想要 | 現實 | 建議方案 |
|---|---|---|
| 凌越改商品 → 網站馬上更新 | 文件無 webhook，需網站主動拉 | 排程 5-10 分鐘同步 + 結帳前再查一次 |
| 凌越改庫存 → 網站馬上更新 | 同上 | 商品頁讀快取（TTL 5-10 分鐘）+ 結帳前 lock 查 `000009` |
| 網站下單 → 凌越馬上看到 | API 支援轉入，但同步呼叫有風險（凌越掛了客戶下不了單）| 網站先成立訂單 → 背景推 ERP → 失敗自動重試 |
| 訂單狀態雙向同步 | 凌越欄位與網站狀態不一定 1-to-1 | 網站維護自己的訂單狀態，凌越做 ERP 單據 |

**會遇到的問題**：
1. 真正秒級即時推播在 207 頁文件裡看不到 → HJ 要接受「秒級～分鐘級延遲」
2. ERP 掛了或網路斷 → 網站要不要擋下單？**建議不擋，背景重送**
3. 庫存：商品頁顯示的庫存是快取，可能不準 → 結帳前再 lock 查最新可用量
4. 預留給連鎖店的庫存：**建議實作為「獨立倉庫」由凌越維護**，網站只查一般倉，不要在網站做預留邏輯

**建議跟 HJ 同步的節奏**：
- 庫存同步頻率：可接受幾分鐘延遲（如 5-10 分），方案最穩、成本最低
- 訂單同步：網站立即成立 + 背景推 ERP（不擋客戶結帳）
- 商品 / 客戶：每日批次 + 異動時觸發
- 真要做秒級即時 → 需另行評估，可能要凌越開 webhook，預算與時程都會跳

## B. 主題分層清單

> 每題格式：🔵 初階（必問）／🟡 進階（簽約後）／給誰／為何這樣分層

### B1. 樣品流程

- 🔵 **初階**：客戶同時加「樣品 + 公版」時，HJ 希望（A）禁止一起結帳、（B）系統自動拆成兩筆、（C）同付款分開出貨？
- 🟡 進階：樣品是否進凌越 ERP？是否扣庫存？樣品運費是固定還是依宅配/超商/材積算？
- 給誰：HJ
- 為何分層：初階是商業決策（A/B/C 選一就好）；進階是選完後才需要的細節

### B2. 私版詢價 + LINE

- 🔵 **初階**：私版詢價單成立後，網站只保留紀錄，等 LINE 確認金額才進凌越 — 這樣對嗎？
- 🟡 進階：LINE 帶詢價單編號要走哪一種技術方案（URL Scheme 預填訊息 / Messaging API push）？訂金支付後才建正式訂單嗎？
- 給誰：HJ（初階）+ 馬亞（進階技術選型）
- 為何分層：初階定流程方向；進階是技術實作細節

### B3. 會員價格類別 ⚠️ **必問**

- 🔵 **初階**：HJ 會員價格類別總共幾種？是否只會落在凌越內建的 5 組（市/大盤/中盤/小盤/一般）？「北部直客 / 中南部直客 / 北部盤商 / 中南部盤商」之外還有嗎？是否會有「單一客戶專屬價」？
- 🟡 進階：若超過 5 組或有專屬價，凌越是否有「客戶別商品價格表」API？資料結構？欄位對照？
- 給誰：HJ（初階）+ 凌越（進階）
- 為何分層：初階決定資料模型複雜度（影響報價與會員系統架構）；進階是凌越技術細節
- ⚠️ **這題初階不問清楚，整個會員系統價格邏輯可能要重做**

### B4. 訂單同步進 ERP

- 🔵 **初階**：未付款訂單是否也要立即進凌越？訂單同步失敗時，HJ 希望（A）直接擋下單、（B）先成立後背景重送、（C）成立後人工補送？
- 🟡 進階：新會員首單時還沒 ERP 客戶編號（CT_NO），HJ 想用哪種策略（先建客戶 / 待綁定佔位 / pending 審核）？訂單同步用 outbox pattern 還是同步呼叫？
- 給誰：HJ（初階）+ 馬亞（進階架構）
- 為何分層：初階是「網站要不要擋客戶結帳」的 UX 決策；進階是失敗處理與架構選型

### B5. 庫存即時更新 ⚠️ **A3 核心題、必問**

- 🔵 **初階**：HJ 能接受的庫存延遲是多少？秒級、分鐘級，還是只要結帳前準確就好？
- 🔵 **初階**：網站前台要顯示「實際庫存數量」，還是「有貨/缺貨」、還是「庫存不足 N 件以下提示」？
- 🟡 進階：凌越是否有庫存異動 webhook / push？預留給連鎖店的庫存能否用「獨立倉庫」實作（建議方案）？
- 給誰：HJ（初階）+ 凌越（進階）
- 為何分層：初階決定架構與成本（秒級即時要凌越開 webhook 才能做）；進階是凌越能力查詢

### B6. 物流 / 超商寄件單 / 材積 ⚠️ **影響報價、必問**

- 🔵 **初階**：四大超商寄件單要不要做後台批次列印？要的話 HJ 目前用哪個平台列印（綠界物流 / 物流商自己平台 / 人工）？
- 🟡 進階：箱購 + 條購混合下單時，材積如何加總？超商 5kg / 45×30×30cm 限制是看單一商品、單一箱、還是整張訂單包裹後尺寸？
- 給誰：HJ
- 為何分層：初階決定「超商寄件單」要不要納入專案範圍（**影響報價甚多**）；進階是運費計算規則
- ⚠️ **超商寄件單凌越文件沒這 API**，若 HJ 期待網站做，要另接綠界物流或其他物流商，必須提早讓 HJ 知道

### B7. 金流 / 發票 / 退款

- 🔵 **初階**：綠界帳號何時申請？由誰申請？**沒帳號就無法開發測試**，會卡時程
- 🔵 **初階**：發票由凌越開立、綠界開立，還是另接電子發票服務（如台灣電子發票服務商）？
- 🟡 進階：信用卡退刷後是否要連動凌越作廢電子發票（imode 第 12 位旗標）？匯款截圖由誰審核？
- 給誰：HJ
- 為何分層：初階卡時程與成本範圍；進階是流程細節

### B8. 商品資料 / 上架

- 🔵 **初階**：商品資料正式來源是凌越、Shopline Excel、新網站後台，還是混合？SEO/商品文案/上下架排程由誰維護？
- 🟡 進階：加購商品規則由誰維護？預約上架/下架是否同步影響凌越商品停用 SK_STOP？
- 給誰：HJ
- 為何分層：初階決定資料同步方向（影響整個架構）；進階是後台功能細節

### B9. 歷史訂單

- 🔵 **初階**：會員中心歷史訂單要顯示哪些欄位（金額/商品/狀態/物流/發票）？舊訂單也要支援「再購買一次」嗎？
- 🟡 進階：要讀凌越 `0000A0` 訂單還是 `0000A1` 銷貨單？舊客戶資料（Email / 手機 / 統編）是否齊全可配對 ERP 客戶編號？
- 給誰：HJ（初階）+ 凌越（進階）
- 為何分層：初階決定畫面範圍；進階是技術實作

### B10. 預留庫存 / 缺貨提醒

- 🔵 **初階**：預留給連鎖店的數量，可不可以用「獨立倉庫」管理（網站只查一般倉，建議方案）？缺貨提醒通知誰（業務 / 倉管 / 客戶）？
- 🟡 進階：缺貨提醒透過 LINE 還是 Email / 站內？提醒門檻怎麼設定？
- 給誰：HJ
- 為何分層：初階是架構選擇；進階是通知設定

### B11. 訂單狀態

- 🔵 **初階**：訂單狀態（待付款 / 已付款 / 備貨中 / 已出貨 / 已完成 / 已取消）由網站維護，還是凌越回寫？
- 🟡 進階：「已完成」7 天後跳轉，起算點是已出貨日 / 物流到貨日 / 人工改為已出貨後第 7 天？訂單修改完全取消還是有條件允許？
- 給誰：HJ
- 為何分層：初階決定 source of truth；進階是 UX 細節

### B12. ATM / 貨到付款 / 自取

- 🔵 **初階**：ATM 虛擬帳號是否確定要做（HJ 回覆「考慮中」）？自取要不要顯示「可取貨」狀態通知客戶？
- 🟡 進階：自取備貨時間如何由商品設定帶到訂單（多商品取最大值 / 逐項顯示）？
- 給誰：HJ
- 為何分層：初階決定範圍；進階是計算邏輯

## C. 給 Wayne 的判斷建議

### C1. 提案階段建議實際問 HJ 的清單（10 題核心）

把所有 🔵 初階題挑出來，**這 10 題務必問**，其他擱置：

| # | 主題 | 一句話題目 |
|---|---|---|
| 1 | 樣品 | 樣品 + 公版同購物車的處理（A 禁止 / B 自動拆 / C 分開出貨） |
| 2 | 私版詢價 | 詢價單成立後只保留紀錄，等 LINE 確認金額才進凌越 — 這樣對嗎？ |
| 3 | 會員價 ⚠️ | 會員價格類別總共幾種？是否只在凌越內建 5 組內？ |
| 4 | 訂單同步 | ERP 同步失敗時要擋下單還是先成立後重送？ |
| 5 | 庫存延遲 ⚠️ | 庫存延遲可接受多久（秒/分鐘/結帳前準確）？ |
| 6 | 庫存顯示 | 顯示實數 / 有貨缺貨 / N 件以下提示？ |
| 7 | 超商寄件單 ⚠️ | 要做後台批次列印嗎？目前用什麼平台？ |
| 8 | 綠界帳號 | 何時申請？由誰申請？ |
| 9 | 發票 | 由誰開立（凌越 / 綠界 / 另接）？ |
| 10 | 訂單狀態 | source of truth 是網站還是凌越？ |

### C2. 提案階段建議問凌越的清單（3 題輕量）

提案階段不要送 80 題給凌越，**會嚇到、影響合作關係**。建議只問：

| # | 一句話題目 | 用途 |
|---|---|---|
| 1 | HJ 帳號目前授權哪些資料類別？是否包含商品 / 客戶 / 訂單 / 庫存 / 收款 / 發票六項？ | 確認可行性，是否有底 |
| 2 | 是否有「客戶別商品價格表」API？（影響會員分級價是否可實作） | 補價格層級的關鍵盲點 |
| 3 | 簽約後可以多快提供測試環境 + XML 範例？預期前置時間？ | 排專案時程 |

剩下 80+ 題壓進「kickoff readiness checklist」，等簽約後啟動會議再交。

### C3. 不該丟客戶 / 凌越，留作內部判斷

以下是馬亞 / 我們內部技術判斷，**提案階段絕對不要丟出去**：

- BFF / Outbox pattern / 三段式庫存（架構選型）
- imode 第 8 位 / OR_SPCHECK / 退刷發票作廢旗標細節
- 整合服務層用 REST 還是 GraphQL
- 訂單編號規則由網站還是凌越主導
- 失敗重試的退避策略（指數退避）
- 金鑰快取細節
- SOAP → REST 包裝細節

這些寫在內部技術備忘 `lyserp-api-reference-index.md` §7 雷區即可，**不對外**。

### C4. Wayne 還要拍板的事

1. 提案階段是否一次給完整 80 題清單，還是先給上面 10 + 3 題？**強烈建議後者**
2. 三批問題的執行順序與時間點（HJ 先 / 凌越同步 / 馬亞自存）
3. 綠界帳號的推進主導權（HJ 自行申請還是我們協助）
4. LINE 詢價方向（A 保底 / B 目標）何時跟 HJ 拍板
5. mockup 凍結期到什麼里程碑（HJ 回 10 題後？簽約後？kickoff 後？）

## 結語

我和 Codex 的清單沒有寫錯，題目品質都好，但**全部一起問會嚇到 HJ 與凌越**，可能造成：
- HJ 覺得「這團隊很麻煩」、「成本一定很高」、「還沒簽約就一堆問題」
- 凌越覺得「客戶還沒確定就要 80 題答案？」可能拖延或敷衍回覆

建議走法：

| 階段 | 問什麼 | 來源 |
|---|---|---|
| 提案中 | §C1（10 題給 HJ）+ §C2（3 題給凌越）| 本檔 |
| 簽約後 kickoff | Codex 完整 80 題給凌越 | `hj-maya-to-lingyue-api-confirmation-items.md` |
| 開發中 | 我的索引 §7 雷區、Outbox / BFF 等技術筆記（內部用）| `lyserp-api-reference-index.md` |

Codex 的完整清單**價值仍在**，作為 kickoff 後的「準備度檢查表」剛剛好。
