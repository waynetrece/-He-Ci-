## 1 · 後台不畫 UI，改畫資料與決策

### 為什麼不示意後台畫面

HJ 已有固定的後台系統。如果我們示意後台畫面，客戶可能會誤會：
- 「你們要取代我們現有後台？」
- 「這套後台是我們收下後要自己用？」
- 「介面跟我們現在的不一樣，是要重學？」

→ 後台**只討論功能與資料流**，不畫 UI。

### 採用方法（兩種並用）

#### a. 資料流圖

用箭頭表示「資料從哪流到哪」，**只畫資料 + 系統節點，不畫 UI**。

範例（訂單成立）：

```
[會員] → 結帳送出
         ↓
      [新網站] 存一份訂單，狀態：待確認
         ↓ (API / 批次同步，待確認)
      [凌越 ERP] 接收，狀態：已建檔
         ↓ (回傳 ack)
      [新網站] 狀態更新：已成立
         ↓ (通知)
      [會員] Email / LINE / 站內訊息（待 Q）
```

> 客戶看完能一眼懂「新網站只是 ERP 的上游／下游某一段，沒取代任何東西」。

#### b. 業務情境清單

表格：「業務在做什麼動作 → 資料流到哪 → 誰是 source of truth → 待確認 Q」。

範例：

| 業務動作 | 資料流向 | Source of Truth | 待 HJ 確認 Q |
|---|---|---|---|
| 業務在 ERP 改客戶等級 | ERP → 網站（單向 push 或定期 pull？） | ERP | 同步頻率、失敗處理 |
| 會員在網站改公司資料 | 網站 → ERP（要審核才同步？） | ERP（最終） | 是否需後台審核才同步 |

> 兩種並用：a 給整體架構印象，b 給具體待確認題；客戶可以直接在 Q 旁邊回覆。

## 3 · 順序建議

### 我們的內部執行順序

1. **先做（畫面）**：購物車 + 結帳 3 頁
   - 購物車 → 結帳（付款 / 物流 / 發票 / 收件地址）→ 訂單成立 / 等待付款 / 月結審核結果
   - 補完前台閉環、自然帶出金流 + 物流問題
   - 不需先選視覺風格，純功能補齊
2. **同步整理（文件）**：ERP 串接資料流圖 + 業務情境清單（#15）
   - 給 HJ 填的核心問題包
3. **接著（文件）**：通知系統事件清單（#9）+ 後台權限矩陣（#14）
4. **後續（畫面）**：商品頁的會員價顯示
   - 等 HJ 回完價格層級題（顧客管理 #4）後再做，避免風格白做

| 包次 | 主題 | 對應 # |
|---|---|---|
| **第 1 包**：交易閉環 | 購物車 / 結帳 / 金流 / 物流 / 發票 | #1, #7, #8 |
| **第 2 包**：ERP 與訂單 | ERP 串接 / 庫存 / 訂單管理 / 顧客等級 / 歷史訂單匯入 / 退換貨 / 樣品後續 | #2, #3, #4, #15, #16, #17, #18 |

## 7 · Codex 審查回覆（2026-04-28）

> 審查範圍：對照 `BackendArchitectureMap.tsx` 12 個後台模組、已完成前台 / 私版 / 會員 / 樣品 mockup，以及 Wayne 前面對「不要畫後台畫面」的要求。

### ✅ 可以

1. **不畫後台 UI 的方向正確。**  
   HJ 既然已有固定後台系統，現在畫一套後台介面很容易讓客戶誤會「要取代既有後台」或「要學一套新系統」。用資料流圖 + 業務情境清單，會比較像在確認串接與責任分工。

2. **12 個後台模組都有接到。**  
   對照 `BackendArchitectureMap.tsx`，公版商品、私版商品、訂單、庫存、顧客、網頁設計、金流、物流、數據分析、行銷工具、網站維護、權限管理都有出現在 §2。

3. **數據分析先聚焦 3 項是對的。**  
   客戶目前只提「商品瀏覽量 / 加購量 / 實購量」，先不要主動延伸到月營業額、報價成交率、會員留存等報表，避免 scope 被放大。

4. **先補購物車 + 結帳閉環合理。**  
   現在公版商品、私版報價、會員都已經能看，但還缺「加入購物車 → 結帳 → 訂單成立」這段，這會自然帶出金流、物流、發票、ERP 建單、會員價等核心問題。

### ⚠ 建議調整

1. **★ 不要定義成「即時雙向同步」。**  
   目前 §2 寫 `★ = 需與凌越 ERP 即時雙向同步`，這句太早承諾。實務上不同資料方向不一樣：
   - 訂單：通常網站 → ERP。
   - 庫存：通常 ERP → 網站，網站只做預留或扣抵請求。
   - 顧客等級：可能 ERP → 網站，也可能網站申請後 ERP 審核。
   - 商品資料：可能 ERP / Excel / 網站後台混合。

   建議改成：

   > **★ = 需與凌越 ERP 串接，實際同步方向、頻率與失敗處理待確認。**

2. **私版商品管理的 Q 稍微太薄。**  
   #6 目前只問客製商品上架、Excel 匯入、預約上架。建議補：
   - 哪些客製商品可即時報價，哪些需人工報價？
   - 客製選項字典由誰維護？
   - 印刷檔案格式 / 大小 / 保存期限？
   - 報價有效期、最低訂購量、階梯價規則？

3. **通知系統要補 Email 基礎條件。**  
   #9 有 LINE / Email / SMS，但建議補：
   - 寄件網域與寄件名稱。
   - SPF / DKIM / DMARC 由誰設定。
   - 通知模板由誰維護。
   - 通知失敗是否要重寄或只留紀錄。

4. **數據分析 3 項要定義口徑。**  
   不用擴大報表，但要問清楚「怎麼算」：
   - 商品瀏覽量：PV、UV，還是會員瀏覽？
   - 加購量：加入購物車次數、商品數量，還是有效購物車？
   - 實購量：付款完成、訂單成立、出貨完成，還是未取消訂單？

5. **順序建議可以分成 3 包給客戶。**  
   §3 的順序合理，但 17 題一次丟給客戶會偏重。建議會議節奏：
   - 第一包：購物車 / 結帳 / 金流 / 物流 / 發票。
   - 第二包：ERP 串接 / 庫存 / 訂單 / 顧客等級。
   - 第三包：通知 / 權限 / 數據 / 維護 / 行銷。

### ❌ 重要遺漏

1. **樣品申請後續處理需要進清單。**  
   已完成樣品申請流程，也討論過會員樣品紀錄，但 §2 沒有單獨接住「樣品申請送出後誰審核、誰出貨、是否占庫存、是否進 ERP」。建議新增或併入 #8 / #2：

   > 樣品申請處理：是否需業務審核、是否限制申請資格 / 次數、是否占庫存、是否產生出貨單、是否進 ERP、樣品物流狀態是否給會員查。

2. **正式環境 / 網域 / 寄信 / 個資法需要至少一題。**  
   這不是畫面功能，但會影響上線與責任邊界。建議加到 #13 網站維護，或獨立成「上線前置條件」：
   - 正式網域與 DNS 由誰管理。
   - SSL、寄信網域、Email 驗證由誰設定。
   - 隱私權政策、服務條款、Cookie / 追蹤同意由誰提供。
   - 會員資料保存、刪除、匯出需求。

3. **資料欄位對照表需要放進 ERP 串接文件。**  
   不建議第一輪跟客戶看太技術的欄位表，但 #15 ERP 串接最終一定需要：
   - 客戶欄位對照。
   - 商品 / SKU 欄位對照。
   - 訂單欄位對照。
   - 庫存欄位對照。
   - 狀態碼對照。

   這不是後台 UI，也不是「既有後台對照」，可以作為資料流圖後的附件。

### 💡 其他建議

1. **「後台不畫畫面」可以保留，但改成「後台不畫 UI，改畫資料與決策」。**  
   這樣語氣比較完整，不會讓客戶以為後台完全不討論。

   - 結帳：付款、物流、發票、收件地址。
