## 3 · 會員系統規劃（待你 review）

### 7 個頁面

| # | 路由 | 用途 |
|---|------|------|
| 1 | `/modules/members/auth` | 註冊／登入 toggle |
| 2 | `/modules/members` | 會員首頁 Dashboard |
| 3 | `/modules/members/orders` | 歷史訂單列表 |
| 4 | `/modules/members/orders/[id]` | 訂單詳情 |
| 5 | `/modules/members/settings` | 帳號設定（含 LINE 綁定）|
| 6 | `/modules/members/quote-list` | 我的詢價單（私版報價的目的地）|
| 7 | `/modules/members/samples` | 樣品申請紀錄（公版樣品的目的地）|

### 點擊流程

```
未登入 → /auth
  ├─ 註冊 (Email) → 填表 → useState 切到「驗證信已寄」 → 點「我已驗證」 → /members
  ├─ 註冊 (LINE) → useState 切到「跳轉 LINE 授權」 → 點「完成」 → /members
  └─ 登入 → /members

/members (Dashboard) — 樞紐
  ├─ 卡：歷史訂單 → /members/orders
  ├─ 卡：詢價單 → /members/quote-list
  ├─ 卡：樣品紀錄 → /members/samples
  ├─ 卡：LINE 綁定狀態 → /members/settings
  ├─ 卡：客戶分級 → /members/settings
  └─ 右上：登出 → /auth

/members/orders → 點訂單列 → /members/orders/[id]
/members/orders/[id] → [一鍵再購] → alert（購物車模組製作中）

/members/settings → [綁定 LINE] → useState：未綁→跳轉中→完成
```

### Dashboard 卡片內容（5 張）

1. **歷史訂單**：最近 3 筆 + 訂單狀態
2. **我的詢價單**：N 筆未送出 / 已送 LINE / 已成交
3. **樣品申請紀錄**：N 筆申請中 / 已寄出
4. **LINE 綁定**：已綁 / 未綁狀態
5. **客戶分級**：VIP / 一般 / 等級價說明

### 訂單詳情頁區塊

- 訂單摘要（訂單號、日期、狀態）
- **配送狀態 timeline**（呼應 PDF「訂單配送狀態」要求）：訂單成立 → 撿貨中 → 已出貨 → 已送達
- 商品明細（品項 + 數量 + 小計）
- 金額明細（商品 + 運費 + 稅 + 總計）
- 收件資訊
- 動作：[一鍵再購] [取消訂單] [退換貨] [追蹤物流]

### 帳號設定區塊

- 基本資料：Email / 手機 / 公司
- **LINE 綁定**（從私版報價搬過來；含完整流程：未綁→跳轉中→完成）
- 通知偏好：訂單通知 / 報價通知 / 行銷
- 安全：改密碼

## 9 · Codex 第三方審查意見（2026-04-28）

### ✅ 可以的部分

- **7 頁架構可以成立**：`/auth`、`/members`、`/orders`、`/orders/[id]`、`/settings`、`/quote-list`、`/samples` 對提案 mockup 來說足夠，不需要一開始再加很多頁。
- **Dashboard 當樞紐是正確方向**：會員中心應把歷史訂單、詢價單、樣品、LINE、會員分級集中呈現，能接上目前公版商品與私版報價模組。
- **LINE 綁定用 useState 模擬足夠**：對提案 mockup 而言，「未綁 → 跳轉中 → 完成」能表達流程，不需要真的串 LINE。
- **Q 列表方向合理**：ERP 客戶編號、分級價、再購、詢價單有效期、樣品狀態都是會影響報價範圍與後續建置的關鍵問題。

### ⚠ 建議調整

- **會員系統定位要從「一般會員中心」改成「B2B 客戶中心」**。畫面上要清楚呈現公司資料、ERP 客戶編號、會員分級價、LINE 聯繫、訂單 / 詢價 / 樣品紀錄，而不是只像一般消費者的帳號頁。
- **Auth 頁要分清楚「LINE 快速登入 / 註冊」與「LINE 帳號綁定」**：
  - `LINE 快速登入 / 註冊`：用來建立或登入會員。
  - `LINE 帳號綁定`：用來接收報價、訂單、出貨通知，或銜接客服。
- **Dashboard 5 張卡建議加入 B2B 身分資訊**：公司名稱、統編、ERP 客戶編號、會員等級、目前套用價格級距。這能讓客戶一眼知道此系統對應他們的 ERP / 分級價需求。
- **「再購一次」建議改成「再訂一次」並做分流**：
  - 公版商品：可加入購物車。
  - 私版 / 客製商品：帶入上次規格，重新詢價或聯絡 LINE 確認。
- **`/settings` 建議命名和內容偏向「帳號與公司資料」**，不要只做一般帳號設定。建議區塊包含基本登入資料、公司資料、發票 / 統編資料、常用收件地址、LINE 綁定、通知偏好。
- **`quote-list` 建議對客戶顯示為「私版詢價紀錄」或「我的詢價紀錄」**，比「詢價單」更容易理解。

### ❌ 重要遺漏

- **同一家公司是否允許多位聯絡人登入**：B2B 客戶可能有老闆、採購、會計、店長。即使這次不做完整權限系統，也應該列為 Q。
- **常用收件地址 / 多門市地址**：餐飲業可能有多門市，會影響訂單配送與再購流程。建議至少在設定頁或訂單詳情頁出現。
- **發票資料**：B2B 場景常會需要公司抬頭、統編、發票寄送方式。可以不用獨立成頁，但設定頁應有區塊。
- **訂單狀態與配送狀態要分開**：訂單可能是待確認、已成立、備貨中、已出貨、已完成、已取消；配送只是其中一段。訂單詳情頁不要只畫物流 timeline。

### 💡 建議補充 Q

| 頁 | 建議新增 / 調整 Q | 原因 |
|----|-------------------|------|
| auth / settings | 同一家公司是否允許多位聯絡人登入？不同聯絡人是否共用同一個 ERP 客戶編號與價格等級？ | 影響 B2B 帳號模型與後續權限設計 |
| auth / settings | LINE 是用來快速登入，還是只用來接收通知與客服聯繫？兩者是否都要支援？ | 避免 LINE 註冊與 LINE 綁定被畫成同一件事 |
| dashboard | Dashboard 是否要顯示公司名稱、統編、ERP 客戶編號、會員等級與價格級距？ | 強化 B2B 客戶中心定位 |
| orders/[id] | 訂單狀態與配送狀態是否分開顯示？狀態名稱與順序由誰提供？ | 訂單流程與物流流程不是同一件事 |
| orders/[id] | 再訂一次遇到私版 / 客製商品時，是直接再下單、重新詢價，還是轉 LINE 確認？ | 客製商品價格與規格可能會變動 |
| settings | 是否需要常用收件地址或多門市地址管理？ | 餐飲客戶可能有多店點配送需求 |
| settings | 發票資料需要哪些欄位？公司抬頭、統編、發票寄送方式是否要讓會員自行維護？ | B2B 場景通常會被客戶問到 |

### 💡 建議開發順序

不建議一次做完 7 頁、3000 行後才驗收。建議先做三個定調頁：

1. `/modules/members/auth`
2. `/modules/members`
3. `/modules/members/orders/[id]`

這三頁會決定會員系統的核心語氣：B2B 身分、ERP 編號、分級價、LINE、再訂流程如何呈現。確認這三頁方向後，再補 `/orders`、`/settings`、`/quote-list`、`/samples`，風險會比較低。

### 文案提醒

- 客戶畫面不要出現 `useState`、`alert` 這類工程語。
- 建議用語：
  - `正在前往 LINE 授權頁`
  - `已完成 LINE 綁定`
  - `此帳號將接收報價與訂單通知`
  - `帶入上次規格，再訂一次`
- 核心定位建議寫成：**會員系統不是單純登入功能，而是 HJ 客戶的公司資料、價格等級、訂單紀錄、詢價紀錄與 LINE 聯繫中心。**

## 10 · 重要前提修正（2026-04-28，由 Wayne 補上）

> **HJ 同時有「一般會員」與「B2B 客戶」兩種使用者**，不是純 B2B。

Codex 的「B2B 客戶中心」定位仍然適用，但要改成：
**同一套系統 + 動態渲染**（不做兩套系統，避免維護分裂）。

### 兩種會員的功能差異初稿

| 區塊 | 個人會員 | 企業客戶 (B2B) |

| 公司資料卡（公司名/統編/ERP/等級/價格級距）| ❌ | ✅ |
| 訂單紀錄 | ✅（精簡）| ✅（含對應公司金額） |

## 16 · 標註不要遮住客戶畫面的視覺 QA（2026-04-28）

- 標註可以框住對應區塊，但 badge / Q pin / 留言泡泡要盡量放在區塊外側，不要壓住主要文字、按鈕、表單欄位、狀態 badge。

- Demo 專用工具列的標註可改放在該列最右側，或只在下方 Q 對照表呈現，不一定要在工具列上放大型 badge。
- 乾淨檢視必須完全不出現標註；標註模式也要保留客戶能看懂原畫面的能力。

### 建議 Claude 調整方式

3. `AnnotationBadge` 可做得更小：例如只顯示圓點 + `建議`，滑過或點開才顯示完整說明。
4. 留言泡泡可與 badge 分開，不一定要緊貼在 badge 旁；避免兩個浮層一起蓋住畫面。

| `Demo 用：點此跳過...` | `提案示意：實際網站會前往 LINE 授權頁` | 比較正式 |

### ❌ 需要修正或確認的風險

#### 1. `npm run lint` 目前沒有過

本機執行 `npm run lint` 失敗，錯在 `src/components/modules/CommentSystem.tsx`：

- `ModalPortal` 裡的 `useEffect(() => setMounted(true), [])`
- `useComments` 裡從 `localStorage` 讀取後同步 `setComments(...)`

錯誤規則是 `react-hooks/set-state-in-effect`。這不一定是 Stage 2 新增頁面造成，但現在如果要宣稱完成，至少要記錄為待修；正式交付前建議修掉。

#### 2. 標註遮擋已有改善，但仍需收尾

Dashboard 頂部遮擋比前一版改善，但仍有幾個點要再看：

- 訂單詳情右上 `Q3` 靠近 `退換貨` 按鈕，仍容易蓋到操作區。
- Dashboard 右上會員等級 badge 附近的 `Q1` 容易貼太近。
- Settings 長頁目前 Q pin 在右側邊角大致可接受，但要確認小螢幕不會蓋住 `新增地址 / 編輯` 操作。

建議原則：Q pin 可標邊角，但不要壓到可點按鈕、狀態文字、表格欄位。

#### 3. Q 文字還是偏長

很多 Q 在主句就塞了太多條件。建議主問題短一點，細節放 context。例如：

目前：

> 訂單狀態與配送狀態是否分兩條 timeline？我們提供以下預設方案，請貴司確認名稱、順序、是否與 ERP / 物流同步：

建議：

> 訂單進度與配送進度是否要分開顯示？

context 再補狀態名稱與 ERP / 物流同步。
