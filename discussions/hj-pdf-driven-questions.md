# HJ PDF-Driven 提問清單

> 日期：2026-04-29
> PDF 來源：`/Users/waynechen/Downloads/HJ網站客製需求表_1150422.pdf`
> 用途：依 HJ 原始需求表結構，把「實作時無法迴避的決策」整理成提問清單。**不從 mockup、不從舊 31 題、不從我們自己想的流程出發**。
> 給 Codex review；不同步 Obsidian。

---

## 寫法原則

1. **完全依 PDF 結構整理**。前台、後台、各區塊照 PDF 順序走。
2. **每個 PDF 條目先引用原文**。
3. **只問「實作時無法迴避的決策」**，不問常識題（例：訂單付款完成才成立，是常規 — 不問，列「合理預設」即可）。
4. **問法白話**，先問現況、再問網站是否照做。
5. **合理預設 vs 可參考做法分開**。一般常規寫「合理預設」，更好的做法寫「可參考做法」，不要混進主問題。
6. **不重複進階備案**。`advanced-features-memo.md` 裡的（月結、自動信用額度、自動拆單、複雜 LINE 推播）不主動放這份。
7. **PDF 沒寫但實作必然遇到的問題**，標「**實作衍生必問**」。
8. **每條目格式**：
   - **PDF 原文**：原始引用
   - **需要確認的原因**：為什麼問這題（給內部看，不傳給窗口）
   - **要問窗口的白話問題**：可直接複製貼 LINE
   - **合理預設**：常規做法，可不問客戶
   - **可參考做法**：更好但客戶要主動接受才做的，不主動推

---

## 優先級標記

- **P1** 核心系統，影響第一版範圍與報價，第一輪先問
- **P2** 商品 / 流程，影響開發工時，第二輪
- **P3** 營運 / 維護，可後送
- **常規** 常識題，不問，列合理預設即可

# 一、前台

## 官網

### 官網 (1) 響應式頁面（RWD）

- **PDF 原文**：「響應式頁面（RWD）」
- **常規**：所有手機 / 平板 / 桌機都要正常顯示，這是現在網站的基本盤。
- **合理預設**：所有頁面 RWD，不另問。

### 官網 (2) 語系：中文 / 英文

- **PDF 原文**：「語系：中文.英文」
- **需要確認的原因**：英文版範圍直接影響翻譯成本與工時。
- **要問窗口的白話問題**：
  > 英文版要做到哪個程度？
  > (A) 全站翻譯（包含商品說明、文案、後台等）
  > (B) 只翻商品名稱與規格欄位
  > (C) 只翻介面，商品內容由 HJ 自行填英文
- **合理預設**：以「介面 + 主商品名」雙語為起點，內文由 HJ 自行決定要不要補英文。
- **可參考做法**：若有國際客戶，可加自動翻譯 plug（成本另計）。

### 官網 (3) 註冊方式：LINE / Email（需驗證）

- **PDF 原文**：「註冊方式：LINE.Email（需驗證）」
- **需要確認的原因**：「LINE 註冊」可能指 LINE Login（要綁 LINE Login Channel），也可能只是「客人有問題用 LINE@ 找我們」。意思差很多。Email 驗證 PDF 已寫，是合理的常規流程。
- **要問窗口的白話問題**：
  > HJ 想做的「LINE 註冊」是指：
  > (A) 客戶可以用 LINE 帳號一鍵註冊新會員（網站要跟 LINE 申請 Login 服務）
  > (B) 註冊還是用 Email，但 LINE 用來客服 / 通知
  > (C) 兩者都做
- **合理預設**：B（Email 註冊 + LINE 客服 / 通知）。LINE Login 可後送。

### 官網 (4) 頁尾不加貴司連結

- **PDF 原文**：「頁尾不加貴司連結（Web Design by…）」
- **合理預設**：頁尾不放開發商 credit。

## 公版商品系列

### (1) 提供給客人下單

- **PDF 原文**：「提供給客人下單」
- **常規**：可下單。

### (2) 樣品申請（每個商品頁增加樣品按鈕）

- **PDF 原文**：「樣品申請：每個商品頁增加樣品按鈕」
- **需要確認的原因**：PDF 沒寫範圍 / 條件 / 後續流程。實作必須決定。
- **要問窗口的白話問題**：
  > 樣品申請：
  > - 哪些商品可以申請樣品？全部 / 只有公版常規 / 部分指定？
  > - 個人會員可以申請嗎？還是只給企業客戶？
  > - 同一個客戶有沒有限制（例：每月最多 5 件 / 每商品 1 次）？
  > - 申請後 HJ 業務要先審核，還是自動寄出？
  > - 樣品要扣庫存嗎？需要產生出貨單、走進凌越嗎？
  > - 樣品免運費還是要付運費？
- **合理預設**：開放申請、業務審核、扣庫存、進凌越（樣品也是出貨）、免運費。
- **可參考做法**：限制次數 / 額度可後台設定。

### (3) 多層規格 & 選項

- **PDF 原文**：「多層 規格&選項」
- **需要確認的原因**：「多層」可能是顏色×尺寸×材質、或是更複雜的組合。每個組合是獨立 SKU 還是合算庫存，影響後台架構與 ERP 對應。
- **要問窗口的白話問題**：
  > 公版商品的規格選項：
  > - 通常會有幾層？（例：尺寸 / 顏色 / 材質）
  > - 每個選項組合是「獨立商品」（凌越裡也是獨立編號），還是「同一個商品多個選項」（庫存合算）？
  > - 不同選項組合的價格 / 庫存會不一樣嗎？
- **合理預設**：每個選項組合 = 獨立 SKU = 凌越獨立客戶編號（PDF「客戶編號相同」邏輯也適用商品編號）。

### (4) 加購商品

- **PDF 原文**：「加購商品」
- **需要確認的原因**：加購邏輯有多種（滿額才能加 / 推薦商品 / 一律可加）。
- **要問窗口的白話問題**：
  > 加購商品的規則：
  > (A) 滿 X 元才能加購某些優惠價商品
  > (B) 一律可加購（在購物車推薦其他商品）
  > (C) 推薦式（看了 A 商品就推薦 B 加購）
  > (D) 結帳前最後一頁推「再加 X 元免運」
  > 加購品的價格，是固定優惠價還是依會員等級？
- **合理預設**：B + D（一律可加 + 結帳前推免運）。

### (5) 離島地區需與客服聯絡下單

- **PDF 原文**：「離島地區需與客服聯絡下單」
- **需要確認的原因**：要決定「擋下時機」。
- **要問窗口的白話問題**：
  > 離島地區（澎湖、金門、馬祖等）的處理：
  > (A) 客戶選收件地址是離島就直接擋下，請客戶用 LINE 聯絡客服
  > (B) 一樣讓他下單，HJ 接單後再聯絡確認運費
  > (C) 離島走另外的固定運費表
- **合理預設**：A（結帳填地址時擋下，導到 LINE@）。

- **PDF 原文**：「API 串接：訂單匯入 ERP」
- **需要確認的原因**：時機、失敗備援、凌越是否有現成 API — PDF 都沒寫，但這是整案最大成本變數。**P1 重點**。
- **要問窗口的白話問題**：
  > 訂單匯入凌越：
  > - HJ 想要訂單多快進凌越？馬上 / 幾分鐘內 / 一天一次?
  > - 萬一資料沒送進凌越（網路斷、凌越維護），訂單還是讓客戶成立嗎？還是擋下來請客戶稍後再試？

- **合理預設**：訂單成立後 5 分鐘內以 API 推給凌越；失敗時訂單仍成立、背景重試 3 次、仍失敗通知 HJ 業務手動補單。
- **可參考做法**：若凌越 API 較弱可走中間轉接層 + 排程同步（5–15 分鐘延遲）；或半自動 Excel 匯出匯入過渡，正式上線後升級。
