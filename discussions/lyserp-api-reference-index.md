## 1. API 整合方式（架構層）

| 項目 | 內容 |
|---|---|
| 主機 | Microsoft Windows Server 2012 以上 + IIS7 以上 + 對外固定 IP |
| 通訊 | Web Service (SOAP with HTTP communication) |
| 流向 | 第三方 ↔ Web Service (API/WCF) ↔ LYERP Database |
| 作業流程 | LYERP 系統開發 → 資料接口規格制定 → 接口內容溝通 → 接口程式撰寫 → 使用者享受便利 |

### 1.1 五個 API 函式（小檔 pp 2-4 / 主檔 pp 1-4）

| 函式 | 用途 | 關鍵傳入 | 回傳 |
|---|---|---|---|
| `LyGetPassKey` | 取得金鑰 | `pusid` 帳號 + `pverifykey` 密碼 | 金鑰字串（成功）；<0 失敗：-1 sql 失敗、-2 帳號不存在、-3 密碼錯、-4 無權限 |
| `LyDataIn` | 轉入（寫入）資料 | `ireno`, `ikye`(金鑰), `icpno`(公司代號), `idakd`(資料種類), `irset`(寫入 XML), `imode`(參數設定 30 BYTES) | 0 成功；<0 失敗（-1 sql 失敗、-2 寫入失敗、-3 金鑰失效（>30 秒）、-4 金鑰不合法、-5 無權限） |
| `LyDataOut` | 轉出（讀取）資料 | `ixmlda`(轉出 XML)、`itmpnm`(分頁暫存檔名)、`itotrec`(總筆數)、`ikye`、`icpno`、`idakd`、`ifld`(抬頭欄位)、`idetfields`(明細欄位)、`irwhere`(過濾式)、`iwhval`(過濾值，多值用 `@#1#@`)、`irec`(每頁筆數，0=不分頁)、`imode`、`iorder` | 0 成功；<0 失敗 |
| `LyDataDel` | 刪除資料 | `ikye`、`icpno`、`idakd`、`ino`(主鍵 / 單號，多值用 `@#1#@` 隔開)、`imode` | — |
| `LyDataPage` | 處理分頁 | `ixmlda`、`ikye`、`icpno`、`idakd`、`ltykd`(0=讀取/1=刪除暫存)、`itmpnm`、`ipageno`(第幾頁) | 0 成功；<0 失敗 |

### 1.2 重要 protocol 細節

- 金鑰 30 秒過期 → 整合層必須短時間內反覆呼叫 `LyGetPassKey`，或於每次調用前重新換金鑰。
- XML 欄位名稱**全部英文大寫**（雖然有些表格示意用小寫，欄位實際以大寫為準；參見 pp 5-6 範例）。
- 文字或備用欄建議用 `<![CDATA[...]]>` 包，避免解譯錯誤。
- `imode` 固定 30 BYTES，每個資料類別自帶配置（pp 詳列每張表的轉入/轉出/刪除參數欄）。
- 過濾式：`irwhere = "no='@v1@' and name='@v2@'"`、`iwhval = "A01@#1#@A02"`。
- XML 結構：`<DocumentElement>` 包多筆 `<LYDATATITLE>`（抬頭）與 `<LYDATADETAIL>`（明細）；同一文件可包多筆抬頭。

### 3.6 發票（0000AS 銷項 IN_*）

關鍵欄位（HJ 站電子發票實作必看）：

- `IN_EINVKD` 雲端發票類別 01~08（07 = 電子發票，這是 B2C 雲端最常用的）
- `IN_EINMOD` 發票開立模式 0=紙本 / 1=B2B / 2=B2C ★必填
- `IN_DONATE` 是否捐贈 0=非/1=捐
- `IN_CARRERTYPE` 載具類別碼：`3J0002`=手機條碼、`CQ0001`=自然人憑證
- `IN_CARRIERID` 載具號碼
- `IN_NPOBAN` 愛心碼
- `IN_RANNO` 發票防偽隨機碼（4 碼數字）
- `IN_B2B_FG` 1=存證 / 2=交換(trunkey)
- `IN_BNO` 統一編號
- `IN_CTNAME` 發票客戶名稱
- `IN_VOIDFG` / `IN_VOIDREASON` / `IN_VOIDDATE` 註銷
- `IN_NOAWARD` 不給獎原因（01~12，常用 04 遺失空白發票、05 銷貨退回作廢、09 金額為零）

## 4. 關鍵代碼速查

### 4.1 計稅 / 稅額計算

- 文件層 `*_TAXKIND`：1=內含, 2=外加, 3=不計算
- 明細層 `*_TAXKD`：1=應稅, 2=免稅
- 商品基本檔 `SK_TAXKD`：同上

### 4.2 客戶/廠商通用

- `CT_PAYFG` 售價種類：1=不設定, 2=零售價, 3=大盤價, 4=中盤價, 5=小盤價, 6=一般價
- `CT_FRM` 發票類別：31=銷項三聯式/電子計算機, 32=二聯式/二聯收銀機, 35=三聯式收銀機, 36=免用, 37=憑證
- `CT_NOPAY` 立帳方式：1=立帳, 2=不立帳, 3=開立發票立帳
- `CT_QKFS` 結帳方式：1=計算天數, 2=固定日期, 3=結帳區間
- `CT_KIND` 國外區分：0=國內, 1=國外, 2=皆是
- `CT_CASH_FG` 收款種類：1=其他, 2=現金, 3=支票, 4=電匯, 5=轉帳, 6=信用卡, 9=承兌交單, A=匯票, B=付款交單, C=L/C, D=信匯
- `CT_TAXKIND`：1=內含, 2=外加, 3=不計算
- `CT_EINMOD`：0=紙本, 1=B2B, 2=B2C

### 4.3 訂單交貨狀態

- 抬頭 `OR_ISPACK`：0=未交完, 1=已交完, 2=交貨確認
- 明細 `OD_IS_PACK`：0=未交完, 1=已交完, 2=交貨確認

### 4.4 配件類別 `*_EQUKIND`

- 0=主件
- 1=標準配件
- 2=選購配件
- 3=贈品配件

### 4.5 驗收狀態（檢驗用）

- `*_CHECKQTY` 驗貨方式：0=免驗, 1=抽驗, 2=全檢
- `*_CHECKOK` / `*_CHECK_FG` 驗收狀態：0=免驗, 1=待驗, 2=合格允收, 3=不合格拒收（`*_CHECK_FG` 多一個 4=不合格拒收/2=驗貨中）

### 4.6 倉庫旗標 `*_WHNO_FG` / `*_SWKIND`

- `WHNO_FG`：0=非寄庫, 1=寄庫
- `SWKIND`：0=廠內倉庫, 1=廠商倉庫, 2=客戶倉

### 4.7 操作模式（idakd 各表的單據單號處理 — 大部分單據共用這組）

| 值 | 含意 |
|---|---|
| 0 | 單號已存在跳過不存檔 |
| 1 | 單號已存在覆蓋 (UPDATE) |
| 2 | 單號已存在按系統編碼重新新增；不存在以目前單號存檔 |
| 3 | 不檢查單號是否存在，直接按系統編碼格式重編單號 |

> 大部分轉入時內定 = 0；HJ 端建單建議用 `0`（避免重複），失敗時讓網站端重試新單號。

### 4.8 imode 第八個位置（很多單據共用）

`imode` 第 8 個位置傳入 `1` 時，該單明細的 `OD_TAX` / `SD_TAX` 欄位變成必填（不再讓系統自動算稅）。

## 7. 雷區與待確認事項（先記下來）

1. **金鑰 30 秒過期** — 整合層需要短時 cache + 自動換鑰機制。
2. **欄位名稱大小寫敏感性** — XML 範例與 5 頁清單顯示「全部英文大寫」，但部分頁的欄位定義出現小寫（`sp_*`/`sd_*`/`md_by1~10` 等）。實作前需向凌越窗口確認大小寫慣例（猜測：實際系統不分大小寫，但 XML 必須大寫）。
3. **OR_NO / SP_NO 編碼權** — 訂單單據單號處理可指定 0/1/2/3 → 建議 HJ 站採 `=0` 跳過存在的，並由網站端決定單號規則（前綴 + 流水）。
4. **發票 IN_RANNO 4 碼隨機** — 雲端發票必填，需要網站端產生，不能空。
5. **商品圖片** — `SK_FIGN` 是檔案名（非 URL），凌越預期放在 `C:\inetpub\wwwroot\ERPAPI\OUTIMG\`（保留原檔名），轉出時 `imode` 第 5 位 = 1 才會帶圖片過來。HJ 站做法可能是：網站圖片 → 拋給凌越時存檔名 / 凌越改圖 → 通知網站重抓。
6. **庫存可用量** — `000009` 提供 `WD_AMTOK`(可用量) 已扣未交訂單，這個適合公版商品頁直接用。但會不會把網站訂單也算進去要看凌越邏輯，建議向窗口確認「網站送 0000A0 訂單後，可用量會即時扣減嗎？還是要等 0000A1 銷貨單才扣？」
7. **私版報價 → 訂單** — 0000A5 → 0000A0 的轉換邏輯由凌越內部，還是要由 HJ 業務手動轉？需確認。
8. **部分表只有轉入或轉出** — 例如 `0000A6 盤點作業` 只能轉入；`000009 目前庫存` 只能轉出（轉出表上標紅 X）。建表階段需注意。
9. **0000A2 銷退單 ↔ 銷貨單發票自動作廢** — 參數 12 控制「轉入退貨單時，如果有對應銷貨單其發票號碼自動處理作廢」，內定 0=否。電子發票在 HJ 站要不要自動作廢需要確認流程。
10. **CT_FRIDFLG 經銷商旗標** — 對應 HJ「經銷會員等級」這條線；經銷商價格由 SK_PRICE2~5 / CT_PAYFG 決定。
11. **金物流與材積判斷分工** — 凌越 API 可提供商品重量 / 材積 / 單位資料，但若另接綠界 ECPay 金物流，金物流主要負責建物流單、列印託運單與貨態通知；超商 / 宅配可配送性仍應由網站先依 HJ 材積、重量、箱購 / 條購規則判斷。問凌越時只確認資料欄位與物流單號回寫欄位，不把配送規則硬推給凌越。詳見 `hj-ecpay-logistics-volume-judgement.md`。
