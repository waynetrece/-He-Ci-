## 1. 一句話結論

窗口回覆已足以把 HJ 電商系統的主要骨架定下來，但還不能直接進入 mockup 或開發。

主要原因：

1. HJ 回覆已確認多個核心業務規則，例如樣品、私版詢價、訂單狀態、庫存來源、金流、物流、會員價。
2. 凌越 API 文件證明 ERP 有商品、客戶、訂單、銷貨、收款、客戶報價、庫存、發票等匯出入能力。
3. 但目前 API 文件偏「凌越通用規格」，還缺 HJ 專案實作必需的環境、權限、欄位對照、實際 XML 範例、狀態碼對應、錯誤處理與同步策略。
4. 所以下一階段應先完成「架構確認 + API / 欄位確認 + 關鍵功能防漏」，再回頭調整 mockup。

## 2. 目前可確認的系統架構

### 2.1 前台公版商品

已確認：

- 公版商品有多層規格：尺寸、容量、顏色、材質、數量（箱、條）。
- 不同規格組合會有不同價格。
- 不同規格組合庫存分開計算。
- 凌越內每個規格組合都有自己的商品編號。
- 加購商品支援兩種情境：
  - 購物車或結帳頁推薦。
  - 買特定商品時出現指定加購品。
- 加購品價格也會依會員等級不同。
- 離島地址不直接結帳，提示聯絡客服。

架構判斷：

- 商品規格應以 SKU / 凌越商品編號作為核心。
- 網站商品資料需要至少包含：商品編號、名稱、分類、規格、會員價格、庫存、SEO、圖片、材積、重量、備貨時間。
- 商品價格不能只做單一售價，必須支援會員/客戶類別價格。

需要再確認：

- Shopline Excel 欄位是否能完整對應新網站欄位。
- SEO、商品圖片、商品描述是否由 Shopline Excel 提供，還是另行整理。
- 加購品規則是由網站後台維護，還是由凌越帶出。

---

### 2.2 樣品申請

已確認：

- 全部公版商品都可以申請樣品。
- 必須登入會員才可成功送出樣品申請。
- 每個品項最多 3 個。
- 最多 10 款品項，總共最多 30 個。
- 樣品免費，但要收運費。
- 樣品不可與販售的公版商品一起出貨。
- Word 內附運費圖：宅配免運門檻 2500，超商免運門檻 699；箱購、條購、箱購+條購的運費規則不同。

架構判斷：

- 樣品應視為獨立流程，不應混入一般商品訂單主流程。
- 可以在前台共用購物車視覺，但結帳時要拆成「樣品申請」與「一般商品訂單」兩種資料。
- 樣品仍需要出貨、運費、會員、庫存或可申請數量管理。

關鍵風險：

- 如果沒有先定義「樣品 + 公版商品」同時加入時的處理方式，後續會補流程、補提示、補訂單拆分規則。

需要再確認：

- 樣品運費金額是否固定，還是依宅配 / 超商 / 材積計算。
- 客戶同時選樣品與公版商品時，是強制拆單、分開結帳，還是同一結帳頁但成立兩筆單。
- 樣品是否要進凌越訂單 / 銷貨單 / 撿貨單，還是只在網站後台管理。

---

### 2.3 私版商品 / 客製詢價

已確認：

- 私版商品用 LINE 溝通細節、傳原始檔、確認下單。
- 網站不需要做檔案上傳、線上校稿、線上確認完稿。
- 網站流程為：
  1. 私版商品頁選擇客製需求。
  2. 成立私版詢價單。
  3. 客人加入官方 LINE，由客服確認製作檔案、金額等。
  4. 雙方確認後支付訂金，下訂成功，開始製作。
- 客戶希望私版商品頁有「上傳檔案」按鈕，但實際是跳轉 LINE 讓客人在 LINE 上傳。
- 客戶詢問是否可以跳轉 LINE 時自動帶出詢價單編號。

架構判斷：

- 私版不是一般購物車商品，應獨立成「詢價單」。
- 網站是詢價單正式資料來源，LINE 是後續溝通渠道。
- LINE 自動帶出詢價編號屬於 LINE OA / 前端跳轉設計問題，不是凌越 ERP API 問題。
- 之後若要把私版詢價轉 ERP，需要決定轉成「客戶報價單」還是正式訂單。

需要再確認：

- 私版詢價單是否要同步進凌越「客戶報價單」。
- 訂金支付後，是否才建立凌越訂單。
- LINE 帶出詢價編號的方式要由瑪雅確認技術做法，並確認 LINE OA 權限與限制。

---

### 2.4 會員、歷史訂單與分級價格

已確認：

- 會員中心要看凌越裡的歷史訂單。
- 再購買一次只適用公版商品。
- 私版客製商品需重新報價。
- 新網站會員先建立，之後再補 ERP 客戶編號。
- 會員價格目前依地區與客戶類型，例如北部直客、中南部直客、北部盤商、中南部盤商。
- 會員等級由業務指定。
- 不同等級價格是每個商品各自設定不同等級價格。

API 判斷：

- 客戶資料 API 有客戶編號、客戶名稱、地址、電話、統編、Email、業務員、客戶類別、地區別、售價種類、等級、授信額度等欄位。
- 商品資料 API 有 5 組價格欄位：市價、大盤價、中盤價、小盤價、一般價，以及輔助單位價格。
- API 清單也提到「客戶貨品項號對照」，但目前完整 PDF 內沒有看到足以直接判斷客戶專屬商品價格規則的完整段落。

關鍵風險：

- HJ 的價格分類可能超過凌越商品基本資料內建 5 組價格。
- 如果實際是「每個客戶 / 每個地區 / 每個商品」都有專屬價，不能只靠 5 組商品價格欄位。
- 歷史訂單要顯示，必須先完成網站會員與 ERP 客戶編號綁定，否則不知道要抓哪個客戶的歷史訂單。

需要再確認：

- HJ 實際價格層級是否只會落在凌越 5 組售價內。
- 若超過 5 組，凌越是否有客戶別價格表或客戶商品對照價格可供 API 匯出。
- 舊客戶資料是否具備 Email / 統編 / 手機，足以與新網站會員配對。
- 歷史訂單要讀「訂單資料」還是「銷貨單」或兩者都要。

---

### 2.5 購物車、訂單與 ERP 同步

已確認：

- 客戶希望網站下單後馬上出現在凌越。
- 凌越有 API 文件。
- 公版與私版不應作為同一購物車訂單處理；私版轉詢價單並由 LINE 客服接單。
- 實際更可能遇到的是「樣品 + 公版商品」一起處理。
- 訂單狀態：
  - 待付款
  - 已付款
  - 備貨中
  - 已出貨
  - 已完成
  - 已取消
- 上述狀態都要讓客戶在會員中心看到。
- 客戶修改訂單功能先取消。

API 判斷：

- 凌越 API 支援訂單資料轉入 / 轉出。
- 訂單抬頭有單據編號、單據日期、交貨日期、客戶編號、客戶訂單單號、付款方式、送貨地址、發票欄位、備用欄位等。
- 訂單明細有貨品編號、品名、規格、單價、單位、數量、折扣、稅額、重量、材積值、倉庫等。

關鍵風險：

- API 有訂單匯入能力，不等於可以直接承諾「網站下單後馬上正確進凌越」。
- 還要確認單號規則、審核狀態、必填欄位、錯誤回傳、重送機制、交易一致性。
- 網站訂單狀態與凌越訂單欄位不一定一對一。

需要再確認：

- 凌越訂單轉入時，單號由網站給，還是凌越自動編號。
- 凌越是否接受未付款訂單先進 ERP。
- 訂單同步失敗時，網站要擋下單、暫存待補送，還是成立網站訂單並通知管理員處理。
- 網站狀態是否存在網站自己的訂單表，凌越只作為 ERP 訂單資料。

### 2.6 庫存管理

已確認：

- 商品庫存以凌越為準。
- 凌越庫存變動後，客戶希望網站馬上更新。
- 預留庫存原因：部分商品會保留給連鎖店，扣掉連鎖店庫存後才能在網路販售。
- 預留庫存只針對特定商品。
- 缺貨提醒通知業務或倉管。
- 缺貨門檻 HJ 可自行設定。
- 客戶詢問是否可以轉到 LINE 上提醒。

API 判斷：

- 凌越 API 支援目前庫存轉出。
- 目前庫存欄位包含貨品編號、貨品名稱、規格、基本單位、倉庫、批號、目前庫存、可用量。
- 文件沒有看到 webhook / push 事件機制。

關鍵風險：

- 「馬上更新」若理解為真正即時推播，目前 API 文件看不出支援。
- 可行做法可能是：
  - 前台查詢時即時向凌越查庫存。
  - 網站排程高頻同步庫存。
  - 網站保留快取，遇到結帳前再次查凌越。
- 預留庫存是 HJ 業務規則，不一定是凌越庫存欄位直接可解。

需要再確認：

- HJ 能接受的庫存延遲是幾秒、幾分鐘，還是必須完全即時。
- 凌越是否有庫存異動推播或只能由網站查詢。
- 預留給連鎖店的數量在凌越哪裡維護。
- 網站前台顯示庫存量、可售量，還是只顯示有貨 / 缺貨。

---

### 2.7 物流、材積與運費

已確認：

- HJ 會自行填入商品材積，需要填入整箱及單條材積。
- 材積用途包含：
  - 判斷能不能超商配送。
  - 計算宅配運費。
- 超過超商限制時，只顯示宅配選擇。
- 四大超商都要支援。
- 超商限制：
  - 尺寸不得超過 45 x 30 x 30 cm。
  - 重量不得超過 5 公斤。
- 多家宅配由 HJ 出貨時填寫，不是客戶下單時選。
- 宅配公司：新航、嘉里大榮、新竹貨運、黑貓、大嘴鳥。
- 客戶查貨態提供物流追蹤連結即可。
- 自取只有一個倉庫。
- 自取需要指定/顯示備貨時間，公版商品頁需填寫備貨時間。

API 判斷：

- 商品 API 有材積、重量、體積、每箱材積、材積單位、外包裝單位、內包裝單位等欄位。
- 訂單明細有材積公式、重量、材積值。
- 凌越 API 文件沒有看到四大超商寄件單 / 宅配託運單 API。

關鍵風險：

- 凌越 API 可以存商品材積與重量，但不代表可以產生超商寄件單。
- 超商批次列印寄件單大概率需要另一套物流服務商 API 或金物流整合，不是目前凌越 API 就能完成。
- HJ 運費圖使用箱購 / 條購 / 箱購+條購規則，網站需要精準定義如何把商品單位與材積加總。

需要再確認：

- 「整箱」與「單條」在商品規格中如何識別。
- 運費圖中的「才」如何由商品材積換算。
- 箱購 + 條購同時下單時，運費規則如何套用。
- 超商寄件單由哪個服務商產生：綠界物流、超商平台、其他物流整合，還是人工上傳。

---

### 2.8 金流、退款與發票

已確認：

- 一般匯款需讓客戶填匯款資料或上傳截圖。
- 綠界預計信用卡，ATM 虛擬帳號考慮中。
- HJ 尚無綠界帳號。
- 宅配貨到付款只限公版商品。
- 自取可現場付款，也可線上信用卡付款。
- 退換貨由 LINE 或電話與客服溝通，客人不得自行申請。
- 退款由客服後台協助：
  - 信用卡由綠界退刷。
  - 匯款由公司會計退回。

API 判斷：

- 凌越 API 有收款單、銷退單、銷項發票、訂單發票相關欄位。
- 但綠界付款、綠界退刷、ATM 虛擬帳號、金流 callback 不屬於凌越 API 文件內容。

關鍵風險：

- 綠界帳號尚未申請，付款與退款流程不能視為已具備。
- ERP 收款 / 發票 / 銷退如何與綠界付款結果同步，需要瑪雅與凌越確認。

需要再確認：

- 綠界帳號由誰申請，何時提供測試與正式商店代號。
- ATM 虛擬帳號是否確定要做。
- 匯款截圖是否要後台審核。
- 發票是由凌越開立、綠界開立，還是網站另接電子發票服務。

## 3. API 文件是否足夠瑪雅直接製作

### 3.1 可以作為開發起點

這份 API 文件足以讓瑪雅知道：

- 凌越使用 SOAP Web Service。
- 有金鑰取得、資料轉入、資料轉出、刪除、分頁等函式。
- 金鑰超過 30 秒會失效。
- 商品、客戶、訂單、銷貨、銷退、收款、客戶報價、庫存、發票等資料種類有匯出入能力。
- 商品、客戶、訂單、庫存等欄位結構大致可查。

### 3.2 尚不足以直接完整開發

目前仍缺：

1. 實際 Web Service endpoint / WSDL。
2. 測試環境 URL。
3. 測試帳號、密碼、公司代號。
4. HJ 啟用哪些資料種類與權限。
5. HJ 實際欄位對照表。
6. 商品、客戶、訂單、庫存的實際 XML 範例。
7. 訂單匯入成功後，凌越回傳單號或主鍵的實際格式。
8. 錯誤碼細節與可重送條件。
9. API 呼叫頻率限制或效能建議。
10. 庫存是否支援即時推播。
11. 狀態碼與網站訂單狀態對照。
12. 價格層級是否足以支援 HJ 的區域與客戶類型價。
13. 超商寄件單與宅配託運單是否另有 API。
14. 發票與金流到底由凌越、綠界或網站哪一方主導。

結論：目前不能說「瑪雅可以照這份直接完成串接」。比較準確是「這份文件可讓瑪雅開始評估，但要請凌越補 HJ 專案實作資料與測試環境」。

## 4. 核心關鍵功能防漏清單

| 核心功能 | 為什麼關鍵 | 目前狀態 | 沒確認的風險 | 要問誰 |
|---|---|---|---|---|
| 樣品與公版商品分流 | 影響購物車、運費、出貨、訂單資料 | 樣品不可與公版一起出貨 | 後續才補拆單 / 提示 / 運費邏輯 | HJ + 瑪雅 |
| 私版詢價與 LINE 銜接 | 影響私版流程與 LINE 介面 | 網站建詢價單，LINE 溝通 | 之後才發現需要詢價成功頁、LINE 編號帶入、狀態追蹤 | HJ + 瑪雅 |
| 訂單即時進凌越 | 影響整體資料主系統 | HJ 要馬上出現在凌越 | API 失敗時訂單資料不一致 | 瑪雅 + 凌越 |
| 庫存即時更新 | 影響前台庫存、結帳防超賣 | 凌越為準，想馬上更新 | 無 webhook 時無法真正即時 | 瑪雅 + 凌越 + HJ |
| 會員分級價 | 影響商品價格與權限 | 業務指定、每商品不同價 | 凌越 5 組價格可能不夠 | HJ + 凌越 |
| 歷史訂單 | 影響會員中心 | 要看凌越歷史訂單 | 無 ERP 客戶編號就查不到 | HJ + 凌越 |
| 材積與運費 | 影響物流選項與結帳金額 | 有材積、超商限制、運費圖 | 箱/條混合計算錯誤 | HJ + 瑪雅 |
| 超商寄件單 | 影響出貨後台 | 四大超商、批次列印 | 凌越 API 看不出支援，需另接物流 | 瑪雅 + 物流/金流商 |
| 綠界金流與退款 | 影響付款、退刷、發票 | HJ 尚無綠界帳號 | 沒測試帳號無法開發驗證 | HJ + 瑪雅 |
